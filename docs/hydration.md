# Hydration 정책 및 의도적 setState-in-effect 설명

이 문서는 프로젝트 전반에서 발견되는 `react-hooks/set-state-in-effect` ESLint 경고가 의도적이며 안전한 이유와, "다크 우선(Dark-first) SSR 정책"에 대해 설명합니다.

## 핵심 요약

- 서버 사이드 렌더링(SSR)은 기본적으로 **다크 테마 우선**으로 렌더링합니다. 이는 초기 HTML/CSS를 일정하게 유지하여 레이아웃/스타일 불일치를 최소화하기 위한 정책입니다.
- 클라이언트에서는 사용자의 실제 테마(로컬 저장소, OS 선호도, 사용자 선택 등)를 읽고 UI를 동기화합니다. 이 동기화는 클라이언트 전용 로직이며, `useEffect` 내부에서 setState를 호출하는 형태로 구현되어 있습니다.
- 결과적으로 라이트 테마 사용자가 페이지를 새로고침하면 최초 렌더(SSR)에서 다크 이미지/스타일이 잠깐 보이고, 클라이언트 마운트 후 실제 라이트 테마로 바뀝니다. 이 깜빡임은 의도된 동작이며, UI/UX 상 허용 범위로 설계되었습니다.

## 왜 setState를 이펙트 내에서 호출하나요?

1. 서버 환경에서는 `window`, `localStorage`, `matchMedia` 같은 브라우저 API에 접근할 수 없습니다. 따라서 초기 렌더는 서버에서 결정된 안전한 기본값(본 프로젝트에서는 `dark`)으로 생성합니다.
2. 클라이언트가 마운트되면 브라우저 API로부터 실제 사용자 선호를 읽고(예: `localStorage.getItem('theme')`, `window.matchMedia`) 상태를 동기화해야 합니다. 이 동기화는 클라이언트 전용이며, 자연스럽게 `useEffect`(또는 의도적으로 `useLayoutEffect`를 선택한 경우도 있음) 내부에서 수행됩니다.
3. ESLint의 `react-hooks/set-state-in-effect` 규칙은 일반적인 케이스에서 성능 이슈를 예방하기 위한 권고입니다. 본 프로젝트에서는 서버-클라이언트 간 역할 분리를 위해 의도적으로 setState를 사용하고 있으며, 이를 문서화하여 유지합니다.

## 다크 우선 SSR 정책

- 목적: 초기 HTML/CSS를 일관되게 유지하여 레이아웃 shift와 스타일 불일치를 줄입니다. 다크 테마를 기본으로 렌더링하면 폰트/이미지/레이아웃이 예측 가능해집니다.
- 적용 범위: `HeroImage`, `ThemeProvider`, `LiveClock`, `i18n-client` 등 클라이언트 상태 동기화가 필요한 컴포넌트.
- 부수 효과: 다크 우선 정책을 따르면 일부 사용자는 페이지 로드 직후 기본 다크 UI를 잠깐 보게 됩니다(그 후 클라이언트가 actual-theme로 전환).

## 깜빡임(다크 → 라이트) 은 의도된 정책인가?

예. 다음 이유로 의도적입니다:

- 서버에서 안전한 기본값(`dark`)을 렌더링한 뒤 클라이언트에서 실제 사용자의 설정을 적용하는 패턴은 SSR 환경에서 흔한 설계입니다.
- 본 프로젝트는 UI 일관성(특히 초기 페인트 시)과 서버 렌더링의 안정성을 우선시하며, 그 대가로 클라이언트 마운트 시점의 빠른 전환(잠깐의 깜빡임)을 수용합니다.

## 운영/유지보수 가이드

- 코드 변경 전: 깜빡임을 없애려면 초기 상태를 클라이언트에서만 계산하거나, `useLayoutEffect`로 paint 전 동기화하는 방법 등이 있으나, 이들은 SSR 가정과 UX에 영향을 줄 수 있으므로 신중히 검토해야 합니다.
- 문서화: 컴포넌트 내부의 주석과 본 문서를 통해 의도를 명확히 유지하세요. ESLint 경고는 자동으로 제거하지 말고, 주석으로 의도를 설명하세요.
